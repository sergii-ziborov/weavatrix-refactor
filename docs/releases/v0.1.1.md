# Weavatrix Refactor 0.1.1

0.1.1 turns rename planning into a complete preview, confirmation, and atomic
apply workflow while pairing the write layer with Weavatrix Core 0.3.13.

## Complete rename workflows

- `rename_symbol` and `rename_related_symbols` now own both phases. The default
  call validates the generated `weavatrix.edit-plan.v1` envelope and returns
  `PREVIEW_OK` with a short-lived, single-use confirmation token.
- Repeating the same operation with `mode="apply"` and that token recomputes the
  deterministic plan, verifies every executable field and working-tree hash,
  then writes atomically with the existing rollback guarantees.
- Rename workflows now carry the explicit `edit` capability. Preview remains
  read-only; apply still requires `WEAVATRIX_ALLOW_SOURCE_EDITS=1`.
- Confirmation fingerprints exclude only generated `createdAt` provenance
  metadata. File hashes, ranges, before/after text, provenance, uncertainty,
  and every other executable field remain token-bound.

## Paired core fix

The minimum core dependency is now `weavatrix ^0.3.13`. Refactor installations
therefore include the corrected `verified_change` duplicate-ratchet state and
direction-preserving `shortest_path` output, together with bounded contract
evidence pagination.
